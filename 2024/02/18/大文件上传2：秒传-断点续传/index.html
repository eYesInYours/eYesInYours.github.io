前言

秒传

秒传的字面意思是：在极短的时间内上传文件。
但这其实是一种障眼法，真实情况是：校验服务端是否已经有这个文件，如有，前端则跳过上传、合并过程，直接显示【上传成功】。

这时注意到一个问题：怎么唯一的标识一个文件？
如果依照文件名，前端所上传的文件若是更改名字后的，那么这个上传依旧要进行吗？

因此，引入文件hash：根据文件内容，唯一地生成一段标识符。

引入spark-md5库

下载后，在项目中有【public】和【src】目录，从【node_modules】中找到<spard-md5.js>，复制到【public】下。

并在【public】下新建<hash.js>。

Worker线程

MDN是这样介绍的：
Web Worker 为 Web 内容在后台线程中运行脚本提供了一种简单的方法。线程可以执行任务而不干扰用户界面。

生成大文件的hash，是一件十分占据线程和时间的事。
如果放在主线程中进行，那么会造成用户界面卡顿、可能无法操作的情况。

可以这样理解Worker线程和主线程间的通信：主线程将生成hash的事交给worker线程，worker线程完成后通知主线程。

补充

<hash.js>和<spark-md5.js>文件为什么要放在【public】目录下？

在以上实例化worker线程时，如果<hash.js>文件是在其他位置，那么worker是没有工作起来的（代码无效）。
原文叙述如下：实例化 web-worker 时，参数是一个 js 文件路径且不能跨域。

可优化

以上所提及的恢复上传，其实仍然是基于本地的。
如果重启浏览器（数据丢失），而又一次进行上传的文件是暂停过的，那么此时进行的上传理应是上传后续未上传文件。

那么，在校验后端文件是否存在的 /verify 接口中，返回结果再加上一种中间状态：

1. 后端已有该文件。
 2. ***后端已有部分***。
 3. 后端没有。

此时后端将已有部分的文件顺序index返回给前端，前端跳过相应文件的上传。
